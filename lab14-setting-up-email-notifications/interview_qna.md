# 🎤 Interview Q&A — Lab 14: Setting Up Email Notifications

## 1) Why are email notifications important in SIEM/SOC monitoring?
Email notifications provide an immediate, low-friction escalation method when alerts trigger. Even if dashboards or UIs are missed, email can notify analysts/admins quickly and support incident response workflows.

---

## 2) What is SMTP and what role does it play here?
SMTP (Simple Mail Transfer Protocol) is the standard protocol used to send email. In this lab, SMTP was used to deliver alert notifications generated by an alert rule/engine.

---

## 3) Why did you use Postfix instead of a public SMTP provider?
The lab used example values like `smtp.example.com`, which are not usable for real testing. A local Postfix relay provides an end-to-end test path in a controlled lab environment without needing external credentials or network dependencies.

---

## 4) What ports are commonly used for SMTP and what do they mean?
- **25**: Traditional SMTP relay (often server-to-server)
- **587**: Submission port (commonly for authenticated client submission, STARTTLS)
- **465**: SMTPS (SMTP over SSL/TLS)

In this lab, **port 25** was used for local relay testing.

---

## 5) How did you confirm Postfix was working?
I verified the service status using:
- `systemctl status postfix --no-pager`

Then I validated mail delivery by sending an alert and confirming it arrived in the local mailbox.

---

## 6) What is the purpose of `/etc/alert_system.conf`?
It represents the SMTP configuration for the alert system, including:
- SMTP host and port
- Credentials placeholders
- TLS enable/disable

In this lab it was configured for `localhost:25` for real local delivery.

---

## 7) What is the purpose of `/etc/alert_rules.conf`?
It defines the alert rule logic in a simple configuration format, including:
- alert name
- condition threshold (5 failed logins within a minute)
- trigger action (send_email)
- recipient

This simulates how alerting rules are stored in many monitoring/SIEM systems.

---

## 8) How did you implement the alert trigger logic?
I created a Bash alert checker script that:
- Counts “Failed password” events in the last 1 minute using `journalctl`
- Falls back to `/var/log/auth.log` if available
- Sends email via `mail` when threshold is met

---

## 9) Why did you use `journalctl` instead of only `/var/log/auth.log`?
On modern Ubuntu systems, authentication events may be accessible via systemd journald. Some systems may also log to `/var/log/auth.log` depending on rsyslog configuration. Using both makes the script more portable.

---

## 10) How did you safely simulate failed login attempts?
Instead of real brute forcing SSH, I generated realistic log entries using:
- `logger -t sshd "Failed password ..."`

This approach avoids security risk and still validates the detection/alert pipeline.

---

## 11) How did you verify the alert condition was met?
I used:
- `journalctl --since "2 minutes ago" | grep "Failed password" | tail -n 5`

This confirmed 5 failed login events occurred within the expected window.

---

## 12) How did you verify email delivery end-to-end?
I checked the local mailbox for the `admin` user:
- `sudo -u admin mail`

Then opened the message and confirmed:
- Subject matches expected alert subject
- Body contains the expected alert message

---

## 13) What would you improve if this was a production system?
Some practical improvements:
- Use authenticated SMTP with TLS for external delivery
- Add rate limiting (avoid alert spam)
- Deduplicate alerts (avoid multiple alerts for the same incident)
- Add structured logging for alert execution
- Integrate with ticketing/on-call tools (PagerDuty, Jira, ServiceNow)

---

## 14) What security risks exist with email alerting?
Common risks:
- Alert content may expose sensitive information (usernames, IPs, internal hostnames)
- Misconfigured SMTP can lead to open relay behavior
- Attackers may attempt to trigger alert storms (noise/fatigue)

Mitigations include sanitizing content, secure SMTP configuration, and alert throttling.

---

## 15) How does this relate to SIEM alerting workflows?
This lab is a simplified version of SIEM alerting:
- SIEM ingests logs (failed logins)
- Detection rule evaluates conditions
- Alert triggers action (email notification)
- Analyst validates and responds

The same idea applies in Wazuh/ELK/Splunk where alert actions can include email, webhooks, and incident automation.
